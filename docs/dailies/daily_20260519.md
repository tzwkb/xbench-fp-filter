## 下一步优化方向讨论

本轮测试结束后，针对"真错误保护率需达到 99.9%"这一核心目标，梳理出以下三个优化方向。

---

### 方向一：恢复原文译文参与匹配 + 补全 reason 标注 + 入库真错误案例

**核心逻辑：** 做数据工程，提升案例库的覆盖度与标注质量。将原文译文重新纳入向量编码，使检索能区分"同一术语对但译文不同"的情况；同时补全现有误报案例的 reason 字段，让 LLM 能够理解每条误报被放行的具体理由；并入库一定规模的真错误案例，使真错误也能被正确召回、形成对比参考。

**优点：** 数据成规模后，检索和判断两个环节都有充分支撑，理论上限高。

**缺点：** 费时费力。需要同时设计误报判定规则和真错误判定规则，标注一致性难以保证；原文译文重新参与编码需要解决字段权重问题，否则会回到长句主导相似度的老问题；且当前 LLM 锚定召回标签的倾向不解决，数据再多效果也有天花板。适合作为**长期并行的数据质量提升方向**。

---

### 方向二：保持术语对匹配 + 大改 prompt 为真错误识别规则

**核心逻辑：** 改变 LLM 的判断框架。现有 prompt 让 LLM"判断是真错误还是误报，两侧都有规则"，导致 LLM 容易找到误报规则为召回案例的标签背书，跟随标签而非独立判断。改造方向是：去掉误报识别逻辑，改为只给出识别真错误的严格条件（即现有 T1/T2/T3 规则），明确说明"满足以下条件则为真错误，否则为误报"。

这样 LLM 不再主动寻找误报理由，召回案例的 review_label 也不再是主导信号；入库案例只需存入误报标签，无需填写 reason，标注成本低。

**优点：** 立即可验证，成本低；不受空 reason 影响；通过严格规则约束 LLM 判断，减少主观发挥空间。

**缺点：** T 规则覆盖是否完整有待验证；去掉软性误报规则（D/E/F）后，部分确实合理的误报可能被误判为真错误，误报识别率可能下降。

---

### 方向三：恢复原文译文参与匹配 + 大改 prompt 为真错误识别规则

**核心逻辑：** 方向二的基础上，同步恢复原文译文参与向量编码。两项同时调整，既通过 prompt 改造提升 LLM 判断独立性，又通过原文译文编码提升检索层对"同术语对不同译文"的区分能力。

**优点：** 检索质量和判断质量同时提升，潜在效果最好。

**缺点：** 引入两个变量，问题定位困难；原文译文参与编码的字段权重问题需要解决，否则退化为初始状态。建议在方向二验证有效后，作为进一步优化叠加。

## Agent 推荐方案

> 以下为 Agent 根据上述三个方向综合分析后给出的推荐，供参考。

**推荐：方向二的加强版——翻转 LLM 举证责任**

在方向二（改 prompt 为真错误识别规则）的基础上，进一步翻转默认判定方向：

- **现有逻辑**：默认倾向误报，T 规则触发才判真错误 → LLM 容易找误报出口
- **改造逻辑**：默认判真错误，只有 LLM 能明确指出命中某条误报规则才放行 → 举证责任转移

同步处理：reason 为空的召回案例不展示 review_label（或附加说明"仅供规则参考，标签不得作为判定依据"），切断 LLM 对空标签的锚定。

**预期效果**：真错误保护率大幅提升（接近 100%），代价是误报识别率下降（部分误报进入人工队列）。由于"漏掉真错误"的代价远高于"多审一条误报"，这个权衡在当前业务目标下是合理的。

**建议执行顺序：**
```
实施方向二加强版（改 prompt + 翻转默认值）
    ↓ 验证真错误保护率是否达标
若达标 → 评估误报识别率损耗是否可接受，配合方向一长期补数据
若不达标 → 叠加方向三（恢复原文译文加权编码）
```
