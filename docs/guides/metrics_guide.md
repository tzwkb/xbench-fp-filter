# 指标解读学习文档
**目标：** 完全理解 RAG 质检复核系统中各项指标的含义、计算方式及综合解读方法

---

## 第一部分：从质检场景出发理解"分类"

### 1.1 系统在做什么

把我们的系统想象成一个"初审员"。AI 初检已经标记了一批疑似翻译错误，我们的系统要对这批条目做二次判断：

- 这条是**误报**（AI 错误报警，实际没问题）→ 系统**压制**，不送人工审核
- 这条是**真错误**（确实有翻译问题）→ 系统**放行**，进入人工审核队列

这是一个典型的**二分类问题**：每条输入只有两种可能的答案。

### 1.2 混淆矩阵：理解所有指标的基础

把所有结果用表格整理出来，就是**混淆矩阵**：

|  | 系统判断：误报（压制） | 系统判断：真错误（放行） |
|---|---|---|
| **实际是误报** | ✓ FA_hit（正确压制） | ✗ FA_miss（漏判误报） |
| **实际是真错误** | ✗ TE_bad（误压真错误） | ✓ TE_safe（正确保留） |

记住这四个格子，所有指标都从这里派生。

用我们项目的数据举例（第七轮，201条）：

|  | 系统压制 | 系统放行 |
|---|---|---|
| **实际是误报（190条）** | 142 条（FA_hit） | 48 条（FA_miss） |
| **实际是真错误（11条）** | 1 条（TE_bad） | 10 条（TE_safe） |

## 第二部分：核心指标逐一解析

### 2.1 误报识别率（Recall）

**公式：** FA_hit / FA = 142 / 190 = **74.7%**

**直觉解释：** 在所有 190 条误报里，系统认出了多少？认出来的才能压制，压制了才能减少人工工作量。

**类比：** 你有一堆文件要分类，其中 190 张是废纸。Recall 告诉你"这堆废纸你扔掉了多少"。

**为什么重要：** Recall 越高，人工需要处理的条目越少，效率越高。

---

### 2.2 精确率（Precision）

**公式：** FA_hit / suppressed = FA_hit / (FA_hit + TE_bad) = 142 / (142+1) = **0.993**

**直觉解释：** 在系统压制的所有条目里，真正是误报的占多少？如果 Precision 低，说明系统乱压制，真错误也被当误报扔掉了。

**类比：** 你扔掉的那堆"废纸"里，真正是废纸的有多少？万一把有用文件当废纸扔了怎么办？

**我们项目的特点：** Precision 全程在 0.96~1.00，说明系统的压制行为非常克制，几乎不会误压真错误。这是这个系统最重要的安全保障之一。

---

### 2.3 F1 分数

**公式：** 2 × Precision × Recall / (Precision + Recall)

**直觉解释：** Precision 和 Recall 往往此消彼长——想识别更多误报（高 Recall），就可能冒险多压制一些，误压真错误（低 Precision）；想确保被压的都是误报（高 Precision），就得保守一点，漏识别一些误报（低 Recall）。F1 是两者的"调和平均"，强迫你同时做好两件事。

**注意：** F1 是对**误报识别能力**的综合评价。它不包含真错误保护率的信息。在我们的项目里，F1 下降（从 0.989 到 0.533）主要是因为 Recall 下降，而 Precision 保持在 1.000，这是主动优先真错误保护的结果，不是系统变差。

---

### 2.4 真错误保护率

**公式：** TE_safe / TE = 10 / 11 = **90.9%**

**直觉解释：** 在所有 11 条真错误里，有多少最终进了人工审核队列（没有被系统错误压制）？

**为什么是第一优先级：** 被系统错误压制的真错误（TE_bad）会绕过所有审核直接流入产品，风险不可控。1 条 TE_bad 的代价远大于 48 条 FA_miss（后者只是增加人工工作量）。

**和 Precision 的关系：** Precision = 1 - TE_bad/suppressed。Precision 高 ↔ TE_bad 少 ↔ 真错误保护率高。两者描述同一件事的不同侧面。

---

### 2.5 整体准确率

**公式：** (FA_hit + TE_safe) / intervened

**更直白的公式：** ≈ (FA/N) × 误报识别率 + (TE/N) × 真错误保护率

代入我们的数据（FA/N = 94.5%，TE/N = 5.5%）：

整体准确率 ≈ 0.945 × 误报识别率 + 0.055 × 真错误保护率

**关键洞察：** 整体准确率被误报识别率主导（权重 94.5%）。真错误保护率再高，对整体准确率的贡献也不超过 5.5%。因此，**整体准确率不适合作为本系统的核心评估指标**，因为它掩盖了最重要的安全维度。

**"全判误报"基准线陷阱：** 如果系统对所有条目都输出"误报"，准确率会有 94.5%——听起来很高，但这意味着 11 条真错误全部流入产品。整体准确率高 ≠ 系统好。

---

### 2.6 待复核率

**公式：** 输出"待复核"或无输出的条目数 / N

**直觉解释：** 系统对这些条目没有信心，放弃判断，默认让它们进人工队列。

**健康水平：** 0~3.5%。待复核率过高说明系统对大量查询无法判断，覆盖能力不足。

---

### 2.7 人工队列压缩率

**公式：** 1 - 人工队列条目数 / N = 1 - (N - suppressed) / N

**直觉解释：** 系统上线前，所有 201 条都要人工审核（工作量 = 100%）。系统上线后，被压制的条目不需要人工看了。第七轮压缩率 71.1%，意味着人工工作量从 201 条降到约 59 条。

**和误报识别率的关系：** 压缩率 ≈ 误报识别率 × (FA/N) + TE_bad × (TE/N)。在 Precision 接近 1.00 的情况下，压缩率 ≈ 误报识别率 × 94.5%。两者几乎等价，只是口径略有不同。

---

### 2.8 审核精度

**公式：** TE_safe / 人工队列条目数

**直觉解释：** 进入人工队列的条目里，真错误占多少比例？

- **未部署系统时：** 11/201 = 5.5%，人工每看 18 条才遇到 1 条真错误，其余 17 条都是误报
- **第七轮：** 10/59 = 17.2%，人工每看 6 条就能遇到 1 条真错误，效率提升 3 倍

**意义：** 审核精度越高，人工审核员越不会"白看"。低审核精度会导致审核疲劳，长期下去反而可能在大量误报中忽略真错误。

## 第三部分：检索层指标

### 3.1 向量相似度是什么

系统在判断一条查询时，会先去案例库里找"历史上有没有类似的错误"。相似度是衡量"多像"的数值，范围 0~1，越高越像。

**技术背景（简化版）：**
- 文字被转换成一个由 1024 个数字组成的"向量"，代表这段文字的语义
- 两段文字的相似度 = 两个向量的"方向相似程度"（余弦相似度）
- 0.7 大致表示"有一定相关性"，0.85+ 表示"非常相似"

---

### 3.2 平均检索相似度

**公式：** 所有查询的 top-1 相似度均值

**我们项目：** 第三轮起固定在 0.695，不随 prompt 变化。

**为什么固定：** 相似度只取决于查询集和案例库的内容，以及把文字转成向量的方式（向量文本）。改变 prompt 规则不影响相似度，只有改变向量文本格式或扩充案例库才能提升它。

**0.695 意味着什么：** 系统大多数时候找到的"最相似案例"只有约 70% 的语义重叠，这是 direct_pass（≥0.80 才触发）长期无法激活的根本原因。

---

### 3.3 三条决策路径

系统根据相似度分三条路走：

| 路径 | 触发条件 | 行为 |
|---|---|---|
| direct_pass | sim ≥ 0.80 且命中误报案例 | 直接压制，不问 LLM |
| llm_review | 0.60 ≤ sim < 0.80，或高相似度命中真错误 | 送给 LLM 加案例参考判断 |
| llm_independent | sim < 0.60 | 送给 LLM 独立判断（无案例） |

**为什么分三条路：** direct_pass 最快最省钱，适合高置信度的明显误报；llm_independent 在找不到参考时退而求其次用纯规则判断；llm_review 是主力，覆盖 96% 以上的查询。

---

### 3.4 LLM 翻转率与翻转准确率

**背景：** 在 llm_review 路径中，系统会模拟"如果纯按检索建议走会怎么判"（这叫 would_mock），再和 LLM 实际输出对比。

**would_mock 的判定规则（注意与路由阈值的区别）：**
- 若无参考案例 → 不判断（不计入翻转统计）
- 若 top 案例 label = 误报 **且 similarity ≥ 0.70** → 模拟判断为误报
- 若 top 案例 label = 真错误 → 模拟判断为真错误
- 否则 → 不判断（不计入翻转统计）

**注意：** would_mock 的相似度门槛（0.70）与路由阈值 THRESHOLD_LOW（0.60）是两套独立逻辑。路由阈值决定"这条查询走哪条路"，would_mock 门槛决定"检索建议是否足够可信到可以拿来作为翻转对比的基准"。0.70 以下的误报命中被视为"建议不够可信"，不参与翻转统计。

**翻转** = LLM 推翻了检索建议，给出了不同的判断

**翻转准确率** = 翻转时，LLM 的判断是否正确

**不翻转准确率** = 没有翻转时，LLM（跟随检索）的判断是否正确

**N/A 的出现条件：** 当某轮全部走 llm_independent（llm_review = 0 条）时，翻转率和翻转准确率均显示 N/A，因为计算分母为 0。第六轮即为此情况（HIGH=0.85、LOW=0.80 导致所有查询相似度均低于 0.80，全走 llm_independent）。

**我们项目的发现（第三、四轮）：**
- 翻转准确率（89~92%）> 不翻转准确率（76~79%）

**结论：** LLM 翻转的时候比不翻转的时候更准，说明 LLM 在主动纠正检索的错误引导。这验证了 RAG 中 LLM 规则层的价值——它不只是在"抄答案"，而是在真正思考。

## 第四部分：如何综合解读多个指标

### 4.1 看指标的顺序

拿到一份测试报告，推荐按以下顺序看：

1. **真错误保护率** — 硬约束是否满足（≥95%）？这是第一眼
2. **误报识别率 + 压缩率** — 系统为人工审核减负了多少？
3. **Precision** — 系统的压制行为可靠吗？（应该 ≥0.95）
4. **审核精度** — 剩给人工的工作"含金量"如何？
5. **LLM 翻转准确率 vs 不翻转准确率** — LLM 的判断质量如何？翻转行为是帮忙还是帮倒忙？
6. **平均相似度 + 相似度分布** — 检索质量如何？有没有改善空间？
7. **整体准确率** — 最后看，仅供参考，不作为主要依据

### 4.2 看指标间的关系

**正常模式（系统健康时）：**
- 误报识别率 ↑ → 真错误保护率 ↓（权衡）
- Precision 维持高位（0.95+）
- 翻转准确率 > 不翻转准确率
- 压缩率 ≈ 误报识别率 × 94.5%

**异常信号（需要关注时）：**
- Precision 突然下降 → 系统开始误压真错误
- 翻转准确率 < 不翻转准确率 → LLM 规则质量下降，翻转在帮倒忙
- 待复核率 > 5% → 系统覆盖能力出问题
- 真错误保护率下降 + 压缩率也下降 → 系统整体失效（不是权衡，是崩溃）

### 4.3 一个快速判断框架

面对任意一轮测试结果，可以用以下问题快速定性：

```
① 真错误保护率 ≥ 95% 吗？
   → 否：当前配置不可用于生产，优先修复
   → 是：继续看下一条

② Precision ≥ 0.95 吗？
   → 否：系统在乱压制，检查阈值设置
   → 是：继续看下一条

③ 误报识别率 ≥ 75% 吗？
   → 否：减负效果有限，寻找改善空间
   → 是：系统达到目标状态

④ 翻转准确率 > 不翻转准确率吗？
   → 是：LLM 在有效纠偏，检索+LLM 组合发挥了作用
   → 否：LLM 规则质量下降，检查 prompt 是否过度调整
```

## 第五部分：延伸阅读建议

如果希望深入理解这些指标背后的理论，推荐以下学习路径（由浅入深）：

### 入门：分类评估基础
- **关键词：** Confusion Matrix, Precision, Recall, F1-score
- **推荐资源：** Google 搜索"Precision Recall 区别"，有大量通俗图解
- **预计时间：** 2 小时

### 进阶：不平衡数据集的评估
- **关键词：** Imbalanced Classification, Class Imbalance Problem
- **核心概念：** 为什么准确率在不平衡数据上会误导，以及 ROC/AUC 曲线
- **与本项目的关联：** 我们的查询集 FA:TE = 190:11，这就是典型的不平衡分类，理解这个概念是读懂我们整体准确率的关键
- **预计时间：** 3 小时

### 进阶：信息检索基础
- **关键词：** Information Retrieval, TF-IDF, Cosine Similarity, Embedding
- **核心概念：** 文字如何被转成向量，相似度如何计算，向量数据库（FAISS）是什么
- **与本项目的关联：** 理解平均相似度、相似度区间分布、direct_pass 机制
- **预计时间：** 4 小时

### 进阶：RAG 系统原理
- **关键词：** Retrieval Augmented Generation, RAG
- **核心概念：** 为什么要把检索和 LLM 结合，各自的局限是什么
- **与本项目的关联：** 理解翻转率、LLM 纠偏行为的意义
- **预计时间：** 3 小时

### 实践：用 Python 自己算一遍
最有效的学习方式是自己动手算。推荐用项目里已有的 LLM JSON 文件（`data/raw_data/test_*.json`），用几行 Python 代码复现每个指标的计算，对照报告里的数字验证。

```python
import json

data = json.load(open("data/raw_data/test_20260422g_llm.json", encoding="utf-8"))

fa_all  = [x for x in data if x.get("ground_truth") == "误报"]
te_all  = [x for x in data if x.get("ground_truth") == "真错误"]
fa_hit  = [x for x in fa_all if x.get("final_label") == "误报"]
te_safe = [x for x in te_all if x.get("final_label") != "误报"]

print(f"误报识别率: {len(fa_hit)}/{len(fa_all)} = {len(fa_hit)/len(fa_all):.1%}")
print(f"真错误保护率: {len(te_safe)}/{len(te_all)} = {len(te_safe)/len(te_all):.1%}")
```

跑通这段代码，再逐步扩展到其他指标，是建立直觉最快的方式。
