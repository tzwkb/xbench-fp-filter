## 系统背景与架构说明

### 业务背景

本系统服务于**游戏本地化翻译质检**场景。游戏文本（对话、技能描述、UI 文字等）在翻译完成后，会经过 AI 质检工具（QA Tool）自动扫描，逐条检测潜在的翻译错误，如术语不一致、漏译、格式错误等。

AI 质检工具的特点是**覆盖率高但误报率也高**——它会大量标记疑似问题，但其中相当一部分并非真正的错误（例如 NPC 对玩家的称呼被合理替换为代词、地名在口语对话中被省略等），这些被过度标记的条目称为**误报**。若将所有标记条目全部送给人工审核员处理，工作量极大且效率低下。

本系统（RAG + LLM 质检复核系统）的定位是在 AI 初检和人工审核之间插入一层**自动二次甄别**，识别并过滤误报，将真正需要人工关注的翻译错误（**真错误**）保留至人工队列。

### 系统定位

```
[原始译文] → [AI 质检工具] → [疑似错误列表]
                                     ↓
                          [RAG + LLM 复核系统]  ← 本系统
                            /       |       \
                      压制误报   保留真错误   无法判断
                            \       |       /
                             [人工审核队列]
                                     ↓
                              [最终质检结论]
```

**系统不替代人工审核，而是作为第一道过滤层，大幅压缩人工需处理的条目数量。**

### 核心术语

| 术语 | 含义 |
|---|---|
| **误报** | AI 初检标记了一条疑似错误，但经过复核确认实际上翻译没有问题 |
| **真错误** | AI 初检标记了一条疑似错误，经复核确认确实存在翻译问题 |
| **压制** | 系统判定某条为误报，从人工队列中移除，不再送给审核员 |
| **放行** | 系统判定某条为真错误（或无法判断），保留至人工队列 |
| **ground_truth（GT）** | 人工标注的正确答案，用于评估系统准确性 |

### 系统架构

系统由三个核心模块组成：

**① 向量检索模块（bge-m3 + FAISS）**

将每条疑似错误的"错误类型 + 错误描述"转换为 1024 维语义向量，在案例库（历史上已人工复核过的错误案例）中检索语义最相似的历史案例。返回 top-k 个候选案例及其相似度分数。

**② 决策路由模块**

根据检索到的最高相似度（top-1 similarity）将每条查询分入三条路径：

| 路径 | 触发条件 | 行为 |
|---|---|---|
| **direct_pass** | 相似度 ≥ 0.80，且命中案例标签为"误报" | 直接压制，不调用 LLM，速度最快 |
| **llm_review** | 0.60 ≤ 相似度 < 0.80，或高相似度命中真错误案例 | 将查询 + 参考案例送给 LLM 判断 |
| **llm_independent** | 相似度 < 0.60（无强匹配） | 送给 LLM 独立判断，不提供案例参考 |

**③ LLM 复核模块（gemini-3.1-pro-preview）**

接收查询条目和参考案例，依据内置的误报判定规则（A-G）和真错误判定规则（T1-T4）给出最终判断："误报"、"真错误"或"待复核"。

**LLM 实际接收的 Prompt 载荷（Payload）：**

LLM 每次调用会收到一个完整的 Prompt，包含以下字段：

| 字段 | 内容 | 来源 |
|---|---|---|
| 错误类型 | 如"术语表违反" | 查询条目 |
| 错误描述 | AI 初检对该错误的描述文本 | 查询条目 |
| 原文 | 被标记的原始游戏文本 | 查询条目 |
| 译文 | 对应的翻译文本 | 查询条目 |
| 历史相似案例（×k条） | 每条包含：错误描述、**人工标签（误报/真错误）**、误报原因 | 案例库检索结果 |
| 判定规则 A-G / T1-T4 | 写死在 Prompt 中的规则文本 | 代码硬编码 |

**关键说明：** LLM 能看到历史案例的**人工判定标签和原因**，这本质上是 Few-shot 学习——LLM 会参考"历史上类似的错误被人工标为误报，原因是 XXX"来辅助当前判断。这是系统设计的核心机制，但也意味着低质量的案例会误导 LLM（第八、九轮的 T 规则实验中，LLM 被 T 规则主导，案例的影响被压制，可以视为 Zero-shot 模式）。

**Prompt 规则速查表（Cheat Sheet）：**

| 规则 | 类型 | 含义 | 版本历史 |
|---|---|---|---|
| **T1** | 真错误 | 术语精确全匹配，但译文完全未体现该术语（非包含/超集关系） | 第7轮引入，沿用至今 |
| **T2** | 真错误 | 官方机构/行政单位名称被简称，导致机构属性丢失（如"开封府"→"Kaifeng"） | 第7轮引入，沿用至今 |
| **T3** | 真错误 | 专有名词拼写与术语表不一致（如 Qin Chuan vs Qinchuan） | 第7轮引入，沿用至今 |
| **T4** | 真错误 | 核心信息完全消失且无任何替代（具体范围经历过扩大和收缩） | 第7轮引入原版；第8轮扩大（加入情感动机条款，导致过严）；第10轮回退+最小补丁 |
| ~~T5~~ | ~~真错误~~ | ~~复合错误类型中任一子问题触发 T 规则即一票否决~~ | 第8轮引入，第9轮废弃（覆盖过宽，误伤大量复合类型误报） |
| **A** | 误报 | NPC 对玩家称呼被译为 you/You，或势力被 them/they 代指 | 初版规则，持续优化 |
| **B** | 误报 | 地名/场所名在口语对话中被 this place/here/the west 等替代 | 初版规则，持续优化 |
| **C** | 误报 | AI 检测的原文字段是术语表词条的子串/超集（如"开封府尹"包含"开封府"），导致错误匹配 | 初版规则 |
| **D** | 误报 | 术语表词条属于 UI/任务类场景，但原文是剧情对话，语境不符 | 初版规则 |
| **E** | 误报 | 译文使用语义等价的同义替换（如 chants 替代 Sect Command） | 初版规则，持续优化 |
| **F** | 误报 | 译文比术语表多了 the/of 等语法词，不影响语义 | 初版规则 |
| **G** | 误报 | 地名/场所名是叙事背景信息，译文聚焦事件/情感而省略，故事信息完整传达 | 第2轮新增 |

**判断优先级：** T 规则 > A-G 规则。一旦触发任意 T 规则，A-G 规则不得覆盖，直接判真错误。

### 案例库

案例库（cases.db + cases.faiss）存储已由人工标注的历史错误案例，每条记录包含错误类型、错误描述、原文、译文、人工复核标签（误报/真错误）及误报原因。

**本次测试时案例库规模：** 110 条（均为产品组标注，已排除合成数据）

案例库是系统检索层的核心依赖。案例库质量和多样性直接决定检索相似度上限，进而影响 direct_pass 路径的触发频率。

### 本次测试概况

- **查询集：** 真实 QA 工具输出的疑似错误条目，共 201 条（误报 190 条 / 真错误 11 条）
- **数据来源：** 5 份人工标注的 QA 报告（xingyun、yaling、余星云、吴雅玲、deepseek），合并去重
- **评估方法：** 将系统判断与人工标注（GT）对比，统计各项指标
- **测试轮次：** 共 10 轮，每轮调整一项或多项参数（阈值、temperature、prompt 规则等）
- **优先级原则：** 真错误保护率为第一优先指标（硬约束 ≥95%），误报识别率为第二优先指标（软约束 ≥75%）

## 指标说明与计算方法

### 三条路径与 final_label 的完整说明

每条查询经系统处理后都会得到一个 **final_label**（最终判断），值为：误报 / 真错误 / 待复核。以下是三条路径各自的行为：

| 路径 | 触发条件 | LLM 参与？ | 有无案例参考 | final_label 来源 | 参与翻转统计？ |
|---|---|---|---|---|---|
| **direct_pass** | 相似度 ≥ 0.80，命中**误报**案例 | 否 | — | 系统直接赋值"**误报**" | 否 |
| **llm_review** | 0.60 ≤ 相似度 < 0.80，或高相似度命中真错误 | 是 | 是（送 top-k 案例） | LLM 判断：误报/真错误/待复核 | **是** |
| **llm_independent** | 相似度 < 0.60（无强匹配） | 是 | 否（无案例参考） | LLM 独立判断：误报/真错误/待复核 | 否 |

**补充说明：**

- **direct_pass 只对"误报"案例触发。** 若高相似度命中的是真错误案例，系统不敢直接压制，会转入 llm_review 交 LLM 判断。
- **llm_independent：LLM 独立判断，不代表系统弃权。** 该路径下系统仍会调用 LLM，在没有案例参考的情况下完全依靠 prompt 规则独立给出判断（final_label 为误报/真错误/待复核）。llm_independent 描述的是"检索层无法提供可信参考"的状态，不是"维持 AI 初检原始结论不变"。
- **llm_independent 不参与翻转统计。** 翻转的定义是"LLM 推翻了检索建议"，但 llm_independent 没有检索到足够可信的案例，无法形成基准对比，因此翻转概念在此路径不适用。翻转率的分母只计 llm_review 条目数。
- **待复核 仅在以下情况出现：** LLM 调用超时、响应解析失败，或 LLM 主动输出"待复核"（genuinely ambiguous cases）。它不是任何路径的默认值，全程占比 0~3.5%。

### 符号定义

| 符号 | 含义 |
|---|---|
| N | 查询集总条数 |
| FA | ground_truth = 误报 的条数 |
| TE | ground_truth = 真错误 的条数 |
| FA_hit | 被系统正确压制的误报（final_label = 误报 且 ground_truth = 误报） |
| TE_safe | 未被系统压制的真错误（final_label ≠ 误报 且 ground_truth = 真错误） |
| TE_bad | 被系统错误压制的真错误（final_label = 误报 且 ground_truth = 真错误） |
| intervened | 系统给出明确判断的条目（final_label ∈ {误报, 真错误}） |
| suppressed | 被系统压制的全部条目（final_label = 误报，包含 FA_hit 和 TE_bad） |
| human_q | 进入人工队列的条目（N − suppressed） |

### 各指标计算公式

| 指标 | 计算公式 | 说明 |
|---|---|---|
| 整体准确率 | \|{x ∈ intervened \| correct}\| / \|intervened\| | 系统有明确判断时的正确比例 |
| 误报识别率（Recall） | FA_hit / FA | 误报中被正确压制的比例 |
| 精确率（Precision） | FA_hit / suppressed | 系统压制的条目中真正是误报的比例 |
| F1 | 2 × Precision × Recall / (Precision + Recall) | 误报识别率与精确率的调和平均 |
| 真错误保护率 | TE_safe / TE | 真错误中未被压制的比例 |
| 待复核率 | \|{x \| final_label = 待复核 或 None}\| / N | 系统弃权比例 |
| 人工队列压缩率 | 1 − human_q / N | 人工工作量减少比例 |
| 审核精度 | TE_safe / human_q | 人工队列中真错误占比（审核信噪比） |
| 平均检索相似度 | mean(top_similarity) | 向量检索 top-1 相似度均值 |
| LLM 翻转率 | flipped / \|llm_review\| | llm_review 路径中 LLM 推翻检索建议的比例 |
| LLM 翻转准确率 | flipped_correct / flipped | 翻转时判断正确的比例 |
| LLM 不翻转准确率 | not_flipped_correct / not_flipped | 跟随检索建议时判断正确的比例 |

### 翻转率详解

**翻转的本质：** LLM 是否推翻了检索建议。

> **工程定位说明：** would_mock **不是**系统线上运行的实际逻辑分支，也不会在生产环境中执行。它是写在 `run_test.py` 的 `_write_report()` 函数中的**离线统计计算**，仅在生成测试报告时运行，用于衡量"如果不调用 LLM、纯靠检索命中标签来决定，准确率会是多少"。新人读代码时不会在线上服务路径中找到 `would_mock`，它只在报告生成阶段存在。

在 llm_review 路径中，系统会用 **would_mock** 模拟"如果完全听从检索命中案例的标签，系统会怎么判"，再与 LLM 的实际输出对比。两者不同 → 翻转，两者相同 → 未翻转。

**would_mock 的判定规则：**

| 检索结果 | would_mock 模拟判断 | 说明 |
|---|---|---|
| 无参考案例 | 待复核（不计入翻转统计） | llm_independent 路径，无法对比 |
| top 案例 label = 误报，且 similarity ≥ 0.70 | 误报 | 高可信度误报命中 |
| top 案例 label = 真错误 | 真错误 | 命中真错误案例 |
| 其他（similarity < 0.70 的误报命中等） | 待复核（不计入翻转统计） | 案例可信度不足，不作为对比基准 |

注意：would_mock 的相似度门槛（0.70）与路由阈值 THRESHOLD_LOW（0.60）是两套独立逻辑。THRESHOLD_LOW 决定"这条查询走哪条路"，0.70 决定"检索建议是否足够可信到能作为翻转统计的基准"。

**翻转统计的覆盖范围说明：**

三条路径都经过了 LLM 判断，但翻转统计只对 llm_review 有意义：

- **direct_pass**：LLM 不参与，无 LLM 判断，翻转概念不适用
- **llm_review**：LLM 参与判断，且 would_mock 能给出非"待复核"的对比基准（相似度 ≥ 0.70 的误报命中，或真错误命中），**可以统计翻转**
- **llm_independent**：LLM 参与判断，但相似度 < 0.60，检索到的案例可信度不足，would_mock 结果永远是"待复核"（无法形成有意义的基准），因此**无法统计翻转**

翻转率的分母是所有 would_mock 给出了明确建议（非"待复核"）的条目数，实际上等于 llm_review 中 would_mock 可对比的那部分。llm_independent 不是没有被 LLM 审核，而是没有可信的检索基准可供对比。

**关键澄清：**

LLM 对 llm_review 路径的**全部**条目都进行了判断，每条都有 final_label（误报/真错误/待复核）。"翻转"和"未翻转"描述的是 LLM 的判断与检索建议是否一致，不代表 LLM 是否作出了判断。

以第三轮为例（llm_review 194 条，would_mock 可对比的共 194 条）：
- **105 条翻转**：LLM 推翻了检索建议（97 条正确，8 条错误）
- **89 条未翻转**：LLM 认同了检索建议，final_label 是误报或真错误，**不是待复核**

### 各指标的业务意义

**整体准确率**：系统在有明确判断时的正确比例。由于查询集中误报占绝大多数（94.5%），该指标受误报识别率主导，仅看整体准确率可能产生误导（例如"全判误报"准确率高达 94.5%，但真错误保护率为 0%）。不宜单独作为核心评估依据。

**误报识别率（Recall）**：直接反映系统为人工审核节省工作量的能力。识别出来的误报才能被压制，未识别的误报仍需人工处理。

**精确率（Precision）**：系统压制的条目中真正是误报的比例。Precision 高说明压制行为可靠，真错误几乎不会被误压。本系统多轮测试 Precision 维持在 0.96~1.00，压制安全性有充分保障。

**F1**：Recall 与 Precision 的调和平均，综合衡量误报识别能力。当两者均高时 F1 高；若优先真错误保护而牺牲 Recall，F1 会随之下降。F1 下降不代表系统变差，可能是主动选择的结果。

**真错误保护率**：系统最核心的安全指标。真错误一旦被误压制将流入产品，修复代价远高于事前拦截。本项目以此为第一优先级硬约束（目标 ≥95%）。

**待复核率**：系统弃权频率，反映覆盖能力。全程维持在 0~3.5%，说明系统对绝大多数查询给出了明确判断。待复核条目默认进入人工队列。

**人工队列压缩率**：最直观的效率指标，直接体现系统对审核员工作量的减少效果。压缩率 = 1 - (人工需处理条目数 / 总条目数)。

**审核精度**：人工队列的"信噪比"。未部署系统时为 TE/N = 5.5%（每 18 条才有 1 条真错误），系统提升该值意味着审核员每查看一批条目能找到更多真错误，减少无效审核。

**平均检索相似度**：检索层质量的代理指标，不随 prompt 变化，仅受案例库和向量文本影响。本次测试固定在 0.695，是 direct_pass 难以触发的结构性原因。

**LLM 翻转率**：翻转率高说明 LLM 在主动纠偏，不完全依赖检索建议。结合翻转准确率来看：翻转准确率高于不翻转准确率（多轮测试中 89~92% vs 76~79%），说明 LLM 推翻检索时的判断反而更准，体现了 LLM 规则层的独立价值。

**整体准确率与各指标的数学关系：**

整体准确率 ≈ (FA/N) × 误报识别率 + (TE/N) × 真错误保护率

= 0.945 × 误报识别率 + 0.055 × 真错误保护率（本次测试数据）

由此可知：提升真错误保护率时，更多误报进入人工队列，压缩率和审核精度均会下降；降低误报识别率时，整体准确率也随之下降。各指标之间存在结构性权衡，需综合解读，不能孤立看单项。

## 优化优先级原则

**核心立场（撰写人个人观点）：** 真错误保护率是系统的首要生命线，误报识别率是效率优化目标，二者不可等量齐观。

### 为什么真错误保护优先

在游戏本地化质检场景中，AI 初检已将所有疑似错误条目标记出来，RAG 系统的任务是对这批条目进行二次甄别——识别其中属于误报的条目并加以压制，真正的翻译错误则需保留至人工复核。

两类决策错误的后果存在本质差异：

- **误报被放行**（系统未能识别误报）：该条目继续进入人工复核队列，增加审核工作量，但不影响最终产品质量，损失是时间成本，可量化、可承受。
- **真错误被压制**（系统将真错误误判为误报）：真实的翻译错误绕过了所有审核环节，直接流入上线产品。这类错误轻则影响玩家体验，重则损害品牌形象，且发现和修复的成本远高于事前拦截。

因此，**宁可多审核一批误报，也绝不放过一个真错误。** 这不是保守主义，而是对质量风险的理性定价。

### 系统价值的正确衡量方式

十轮测试数据表明，整体准确率不是评估本系统的合适指标。原因在于，测试集中误报占比高达 94.5%（190/201），任何偏向"全判误报"的策略都会在准确率上显得漂亮，但这是数据分布带来的统计假象，与系统的实际价值无关。

更合理的参照是**与"不部署系统"相比的增益**：

| 场景 | 人工需审核条目数 | 真错误保护情况 |
|---|---|---|
| 不部署系统（全量人工审核） | 201 条（100%） | 100%（人工兜底） |
| 部署系统（第七轮配置） | 约 59 条（29%） | 90.9%（10/11 真错误保留至人工） |
| 部署系统（第十轮配置） | 约 103 条（51%） | 100%（所有真错误保留至人工） |

以第七轮为例：系统将人工需处理的条目从 201 条压缩至约 59 条，**审核工作量减少 71%**，同时保住了 10/11 条真错误。这是系统真实的效率价值所在。

需要说明的是，较低的整体准确率并不等于系统表现差。当系统向"严格保护真错误"方向调优时，会有更多误报通过系统进入人工队列（拉低准确率），但这恰恰是优先级原则的主动选择，而非系统缺陷。

### 调优目标

基于上述原则，后续优化以以下指标为约束：

- **真错误保护率 ≥95%**（硬约束，不可妥协）
- **误报识别率 ≥75%**（软约束，在不损害硬约束的前提下尽量提升）

当两项指标冲突时，优先保真错误保护率。
