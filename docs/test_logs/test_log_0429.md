## 指标说明

| 指标 | 计算公式 | 说明 |
|---|---|---|
| 整体准确率 | (FA_hit + TE_hit) / N | 系统有明确判断时的正确比例 |
| 误报识别率（Recall） | FA_hit / FA | 误报中被正确压制的比例 |
| 精确率（Precision） | FA_hit / suppressed | 压制条目中真正误报的比例 |
| F1 | 2PR/(P+R) | 误报识别综合指标 |
| F2 | 5PR/(4P+R) | β=2，更重视 Recall |
| 真错误保护率 | TE_hit / TE | 真错误中未被误压制的比例 |
| 待复核率 | 待复核或 None / N | 系统弃权比例 |
| 人工队列压缩率 | 1 - human_q / N | 工作量减少比例 |
| 平均检索相似度 | mean(top_similarity) | 检索质量代理指标 |

**优先级：** TE 保护率 ≥85%（硬约束），FA 召回率 ≥85%（目标），F2 ≥0.80（目标）。

## 关键问题分析

### 1. Prompt 调优 ROI 为零

0429c/d 证明：prompt 的边际调整只能在"偏保守"和"偏宽松"之间摆动，无法突破 ceiling。LLM 在模糊案例上的判断受限于 few-shot 质量，而非规则描述细节。

### 2. reranker 引入 reason bias

0429b/g 中，reranker 改变了 59% 查询的 top-3 内容。当 similarity≥0.85 时 label 被隐藏，LLM 只能根据 reason 推断，而 rerank 后的 reason 来源方向不一定与当前查询一致。

### 3. st+tt 是有效改进

加入 source_text + target_text 后，检索能区分：
- "孤云"→"Guyun Pavilion"（真错误，专有名词形式错）
- "孤云"→"That group"（误报，代词替代）

这直接减少了同术语冲突导致的误判。

### 4. 数据冲突是根本瓶颈

case library 与 query set 存在高 sim 标签冲突，限制 TE 保护率上限。22 个 case library label 修正 + 2 个 query GT 修正后仍有残余冲突。

## 关键问题分析

### 1. Prompt 调优 ROI 为零，但固定 few-shot 有效

0429c/d/h 证明：prompt 的边际调整只能在"偏保守"和"偏宽松"之间摆动，无法突破 ceiling。

**但 0429j 证明：固定 few-shot 是例外。** 8 条代表性误报案例作为固定参考，直接让 FA 从 52% 提升到 76%，且 TE 仍保持 88%。这验证了 LLM 的判断瓶颈不在规则理解，而在**案例参考质量**。

**场景分类（0429h）的教训：** LLM 无法可靠区分正式/非正式场景，取消固定优先级后 A-G 规则被滥用。人对场景的识别远强于 LLM，场景分类对人工标注有价值，但对 LLM 自动判定不适用。

### 2. reranker 引入 reason bias

0429b/g 中，reranker 改变了 59% 查询的 top-3 内容。当 similarity≥0.85 时 label 被隐藏，LLM 只能根据 reason 推断，而 rerank 后的 reason 来源方向不一定与当前查询一致。

### 3. st+tt 是有效改进

加入 source_text + target_text 后，检索能区分：
- "孤云"→"Guyun Pavilion"（真错误，专有名词形式错）
- "孤云"→"That group"（误报，代词替代）

这直接减少了同术语冲突导致的误判。

### 4. LLM 超时问题

Prompt 变长后超时率显著上升。0429j 的解决方案：
- `llm_review.py` 新增 `_MAX_RETRIES = 3` 自动重试机制
- 超时/空响应时等待 2 秒后重试
- 重试后 23 条全部成功，待复核率从 14.5% 降至 0%
