## 第一组：prompt 翻转默认值（wlj-5.20a）

**目的：** 验证将 LLM 默认判定方向从"倾向误报"翻转为"默认真错误，必须命中规则才能放行"后，真错误保护率的提升幅度及误报识别率的实际损耗。

**改动：**
- 删除误报软规则 D/E/F（应用语境有误、同义替换、冠词介词差异）
- 默认判定为真错误，只有明确命中 A/B/C 才改判误报
- 复核步骤第 1 条从"优先采信历史案例"改为"案例标签不得替代规则判断"

**结果：** 真错误保护率从 47.8% 跳至 92.2%（+44.4pp），误报识别率从 99.3% 跌至 45.5%（-53.8pp）。方向完全正确，代价在预期内。

残余 7 条真错误漏报根因：规则 C 被用于推断"游戏规范译名"——LLM 参考案例译文（相似度 1.0）主张术语表词条有误，触发 C 规则放行。

## 第二组：收窄规则 C + 切断空 reason 锚定（wlj-5.20b）

**目的：** 针对 20a 残余 7 条漏报的根因——LLM 利用案例内容推断"游戏规范译名"——加入显式禁止指令。

**改动：**
- 规则 C 明确限定：只能基于 error_description 字段本身的子串/超集关系，禁止通过案例译文推断规范译名
- 历史案例使用规则补充：案例译文不能作为推断"游戏规范译名"的依据

**结果：** 误报识别率小幅提升至 50.4%，但真错误保护率未变（仍 83/90）。说明文字指令无法阻止 LLM 在高相似度（sim=1.0）案例下的锚定行为——7 条漏报全部出现在相似度极高的案例场景，指令被完全忽略。

**结论：** 文字层面的禁令对高置信度案例锚定无效，必须从架构层面切断案例传入。

## 第七组：C③ 扩例 + JSON 字段顺序翻转（wlj-5.20g）

**目的：** 针对报告明细发现的两类根因：① C③ 示例过窄（只有名词复合，LLM 不识别动词复合、颜色词、人名字符等）；② LLM JSON 输出自相矛盾（分析里说"命中C③判误报"，但 final_label 已在前面生成为真错误，无法回头修改）。

**改动：**
- C③ 补充覆盖所有复合词类型的具名示例：动词复合（解→解锁/解救/解除）、形容词复合（绝→绝世外观）、颜色词（紫→紫色、明→夜明珠/查明）、词素字（极→蹦极、皮→皮影）、人名字符（钱→钱多多）
- D 规则补三条具名示例：紫色花卉、遇着太多人、主动交互
- JSON 字段顺序改为 `llm_analysis` 在前、`final_label` 在后，加"先写判定理由，再给出标签"指令

**结果：** 误报识别率从 31.0% 跳至 **71.6%**（+40.6pp），真错误保护率降至 94.4%（漏 5 条）。JSON 顺序翻转是主要功臣——原先大批 C③ 案例 LLM 已经推导正确但标签写错，改顺序后全部修正。C③ 补例覆盖了动词/颜色词等新类型集群。

5 条真错误漏报根因：[122] C③ 误触发专有名词+类型词组合、[285] F 规则误触发作品名介词、[161] D 规则误触发人名字符、[71] C③ 示例引发幻觉、[179] D 规则语境边界判断。

## Agent 总结（测试方式与核心结论）

### 当前架构

```
查询 → 向量检索 → direct_pass（sim≥0.99，直接判误报）
                 → LLM 复核（search_results=[]，纯规则判断）
```

**案例库对 LLM 完全不可见。** 向量检索结果只用于 direct_pass 路径。进入 LLM 时传空列表，LLM 只依据 A-F 规则独立判断。当前阈值 0.99 极严，358 条查询中 direct_pass=0，全部走 LLM。

### 关键发现：不传案例反而更好

| | 传案例最优（5.20b） | 不传案例最优（5.20h） |
|---|---|---|
| 误报识别率 | 50.4% | **66.0%** |
| 真错误保护率 | 92.2% | **97.8%** |

案例库中 5131 条误报均无 reason 字段，LLM 拿到的是裸标签。高相似度（sim≥0.97）下 LLM 会被标签锚定，规则判断被架空，文字禁令无效。切断案例传入后，两个指标均优于传案例的历史最高值。

**案例库有价值的唯一场景是 direct_pass（向量相似度极高时直接压制），但当前阈值 0.99 几乎不触发。**

### 案例传入的价值边界

**裸标签案例传给 LLM 是负收益，有 reason 的案例是正收益。**

不传案例优于乱传案例，但这个结论有条件边界：

| 条件 | 不传更好 | 传更好 |
|---|---|---|
| 案例以无 reason 裸标签为主 | ✓ | |
| 案例质量参差、批量入库 | ✓ | |
| 案例全部有高质量 reason | | ✓ |
| 规则尚未充分迭代 | | ✓（案例 50.4% > 规则 20.5%） |

更准确的结论：**案例的价值不在于数量，而在于有没有解释（reason）。** 数据库不完善时，与其传裸标签让 LLM 锚定，不如不传让它独立推理。后续若能对案例库补充高质量 reason，选择性传入有 reason 的案例有望在当前规则基础上进一步提升误报识别率。

### 规则迭代方法论

1. 跑测试 → 读报告明细 → 分析漏报根因 → 补规则 → 再跑
2. 规则靠数据驱动，不靠人工预设
3. 单条规则的具名示例比抽象描述有效得多（LLM 不执行抽象禁令，执行具名例子）
4. JSON 字段顺序影响 LLM 推理：`final_label` 在前时 LLM 先定标签再分析，导致分析与标签自相矛盾；改为 `llm_analysis` 在前后，自相矛盾消失，单次修复带来 +40pp 收益

### 当前规则层天花板

纯规则可覆盖：C 类误匹配（子串/复合词）、D 类语境不符、E/F 边界案例。
无法覆盖：需要游戏领域知识的语义判断（特定场景下的同义词、角色专属译法等）。预计规则层误报识别上限 72-78%，超出需引入外部知识或标注数据。
